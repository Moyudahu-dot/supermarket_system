# 身份认证安全设计与实施说明

## 1. 建设目标

本模块在现有超市管理系统用户表和 Flask Session 机制基础上，补强登录、登出、注册、密码存储和登录风控能力，目标如下：

- 防止用户密码以明文形式保存到数据库。
- 降低暴力破解、撞库和重复试错登录的成功概率。
- 通过简单验证码提高异常登录的交互成本。
- 使用服务端签名 Session 记录登录态、用户角色和会话有效期。
- 为登录、登出、注册接口形成可说明、可演示的安全闭环。

## 2. 密码加密存储

系统使用 Werkzeug 的 `generate_password_hash` 生成带盐密码摘要，配置方法为 `pbkdf2:sha256`，盐长度为 16。新增用户和修改密码时，系统不再保存明文密码，而是保存哈希后的字符串。

为了兼容历史数据，登录校验和修改密码流程会先判断数据库中密码是否已经是哈希格式：

1. 如果是哈希格式，则使用 `check_password_hash` 验证密码。
2. 如果是旧版明文格式，则仅在本次校验通过后允许登录，并立即将该密码升级为带盐哈希值。
3. 所有新注册、管理员新增用户、修改密码产生的新密码均直接保存为带盐 SHA-256 派生哈希。

该方案避免了数据库泄露后攻击者直接获取用户原始密码，同时保留了对旧数据的平滑迁移能力。

## 3. 登录失败次数限制

系统按用户名维护登录风险状态，记录连续失败次数和临时锁定时间：

- 连续失败达到 3 次后，登录页展示验证码并要求用户输入。
- 连续失败达到 5 次后，账号进入 5 分钟临时锁定状态。
- 登录成功后清除该用户名的失败计数。
- 非 active 状态账号即使密码正确，也不会建立登录会话，并计入失败风险。

当前实现使用进程内字典保存风险状态，适合课程项目和单进程演示；生产环境建议迁移到数据库或 Redis，以便多进程共享状态并支持审计查询。

## 4. 验证码与简单风控

验证码采用四位数字形式，在用户连续失败达到阈值后才出现。设计原因如下：

- 正常用户首次登录不需要额外步骤，降低使用成本。
- 失败次数较多时提高自动化攻击成本。
- 验证码保存在 Session 中，提交时与用户输入进行比对。
- 验证失败会刷新验证码，避免攻击者重复使用同一个验证码。

该验证码属于简单风控，主要用于课程系统演示。生产环境可替换为图片验证码、短信/邮箱二次验证或基于 IP、设备指纹、地理位置的风险评分。

## 5. Session 会话管理

系统使用 Flask Session 保存登录态，主要字段包括：

- `user_id`：当前登录用户 ID。
- `username`：当前登录用户名。
- `role`：当前登录用户角色，用于角色访问控制。

会话安全配置包括：

- `SESSION_COOKIE_HTTPONLY=True`：禁止客户端脚本读取 Cookie，降低 XSS 窃取风险。
- `SESSION_COOKIE_SAMESITE="Lax"`：降低跨站请求携带 Cookie 的风险。
- `PERMANENT_SESSION_LIFETIME=30 minutes`：限制登录态有效期，降低长期会话被盗用的风险。
- 登录成功后先 `session.clear()` 再写入新的登录态，降低会话固定攻击风险。
- 登出时写入审计日志并清空 Session。

## 6. 登录、登出、注册接口

### 登录接口 `/login`

- 支持 GET 展示登录页、POST 提交用户名和密码。
- 登录前检查当前用户名是否处于临时锁定状态。
- 达到验证码阈值后要求校验验证码。
- 密码校验成功后检查账号状态，只有 active 用户可以登录。
- 登录成功后写入 Session，并按角色跳转到对应页面。
- 登录成功写入审计日志。

### 登出接口 `/logout`

- 已登录用户登出时写入审计日志。
- 清空 Session 中的用户身份和角色信息。
- 跳转回登录页。

### 注册接口 `/register`

- 支持 GET 展示注册页、POST 提交注册信息。
- 校验用户名、密码和确认密码。
- 注册用户默认角色为 cashier，避免普通用户自行注册成为管理员或经理。
- 密码在写入数据库前进行带盐哈希处理。
- 注册成功后跳转登录页。

## 7. 权限控制补强

系统继续采用基于角色的访问控制：

- admin：用户管理、审计日志。
- manager：商品和销售统计管理。
- cashier：销售和订单相关功能。

本次补强同时为用户新增、更新和删除接口增加了 admin 角色校验，避免未授权用户直接调用后端接口修改用户数据。

## 8. 后续改进建议

- 将登录失败状态迁移到数据库或 Redis，保证多进程部署下计数一致。
- 增加唯一用户名约束和注册异常处理，避免重复用户名导致数据库错误。
- 在生产环境使用强随机 `SECRET_KEY`，并通过环境变量注入。
- 开启 HTTPS 后设置 `SESSION_COOKIE_SECURE=True`。
- 为修改用户、删除用户等敏感 POST/GET 操作增加 CSRF 防护。
- 增加密码复杂度策略，例如最小长度、大小写字母、数字和特殊字符要求。
- 对登录、注册、改密失败事件写入安全审计日志，便于事后追踪。
